## 📄 Введение ##

---
**TOASTER** - кластерный чат-бот для социальной сети VK.
Он был создан и спроектирован для работы штата модерации в кластере бесед   неформального фанатского сообщества [FUNCKA](https://vk.com/funcka) по игре [STALCRAFT](https://vk.com/stalcraft_official). 
Проект является полностью opensource, что значит - вы сможете использовать бота на своих кластерах бесед.

Получить репозиторий можно, скачав его из списка [релизов](https://github.com/STALCRAFT-FUNCKA/TOASTER/releases) в виде архива.
Пользователи Linux систем могут добиться эквивалентного результата при помощи:

```bash
git clone https://github.com/STALCRAFT-FUNCKA/TOASTER
```

Отлично! Теперь все нужные ресурсы (или почти все) находятся на вашем компьютере. Самое время прейти непосредственно к настройке бота.

## ⚙️ Настройка Config ##

---
**TOASTER**, как и любой другой бот vk, работает из-под капота какого-то сообщества. Поэтому, что бот где-то мог существовать, ему нужно настроить его сообщество, к которому он будет привязан, а затем скомпоновать config файл настроек.

Для начала создадим сообщество.

> vk.com -> Сообщества -> Создать сообщество

Выбираем тип **"Сообщество по интересам"** и называем нашу группу как хотим, это зависит только от вашего вкуса. 

Как только наше сообщество будет создано - вам нужно будет создать в нем чаты, в которых будет действовать бот. Для этого перейдем в раздел настроек:

> Сообщество -> Управление -> Чаты -> Создать чат

Изначально __***необходимо***__ создать как минимум две беседы, одна из которых станет лог-чатом, а другая - первой модерируемой беседой.
Если же вы не планируете создавать в сообществе свои чаты, а пригласить сообщество в другие беседы, используя его там в качестве модератора, то вам нужно совершить следующие действия:

> Сообщество -> Управление -> Сообщения -> Настройки для бота

В этом разделе __***необходимо в обязательном порядке***__ поставить галочку в графе **"Разрешить добавлять сообщество в чаты"**. Это позволит пригласить бота в уже созданный кластер бесед, который будет находиться либо под пользователями, либо под другим сообществом. Напоминание: Даже в случае приглашения сообщества с ботом в сторонние беседы - должно существовать как минимум две беседы.

И так, с целевым использованием бота мы определились, осталось разобраться с файлом Config. Для этого перейдем в скачанный ранее репозиторий, после чего откроем файл config.py.

Внутри нас будут ждать всего три графы, в которые необходимо что-то вставить:
- TOKEN - переменная, которая будет хранить в себе токен бота
- GROUP_ID - переменная, которая будет хранить в себе id сообщества
- STUFF_ADMIN_ID - переменная, которая будет хранить в себе id главного админа

Первые две переменные нужны для подключения бота к вашему сообществу, а последняя - для супер-доступа какого-то пользователя ко всем командам бота. Короче говоря, супер-админ.

В переменную STUFF_ADMIN_ID можно вписать id своей страницы или страницы любого другого человека, которого вы хотите сделать супер-админом.
```python
STUFF_ADMIN_ID = 10000001
```

Теперь остается получить токен и id группы, начнем по порядку.

>Сообщество -> Управление -> Работа с API -> СОЗДАТЬ КЛЮЧ <br/>

Боту необходимо выдать все права, иначе работать он не будет. Копируем полученную и вставляем в Config.py, в переменную TOKEN.

```python
TOKEN = "some token abstract key"
```

Укажем и целочисленное значение ID сообщества, на котором будет стоять бот. Копируем и вставляем в переменную GROUP в файле __Config.py__. 
```python
GROUP = 1000000001
```

После чего необходимо обозначить доступ и версию для LongPoll API, для этого переходим сюда: 

>Сообщество -> Управление -> Работа с API -> Long Poll API 

Включаем LongPoll API, выставляем версию __5.131__. Далее переходим во вкладку типов событий и выставляем все права для сообщений. 

**Примечание:** Если вы используете бота в сторонних от корневого сообщество беседах,  то боту нужно выдать доступ ко всей переписке и назначить администратором в беседе. Сделать это можно через список участников беседы

## 💽 Настройка VPS ##

---
В нашем распоряжении имеется репозиторий, с готовым config.py файлом, в котором есть все нужные данные для ввода бота в работу. Сообщество создано, беседы есть - дальше только VPS. Рассмотрим самый простой пример с хостингом [beget](https://beget.com/ru).

Заходим на хостинг, регистрируем новый аккаунт и выбираем вкладку [VPS](https://beget.com/ru/vps). После чего выбираем интересующий нас тариф, рекомендуется взять самый простой. Для TOASTER параметров *(1 core CPU, 1 gb RAM, 10 gb MEM)* данного тарифа вполне хватает.

Дальше пути установки могут разделиться, это зависит лишь от того, что вы предпочитаете использовать. На ваш выбор имеется:
- Готовое решение [VPS на Ubuntu](#-установка-ubuntu) (Для тех, кто вообще ничего не понимает)
- Готовое решение [VPS на Ubuntu с Docker](#-установка-docker) (Для тех, кто уже смешарик)

Вне зависимости от вашего решения - выбираем и создаем VPS, а дальше дело за малым. Остается только дождаться конфигурации VPS и ввода его в эксплуатацию. О состоянии VPS можно будет узнать из панели управления, в которую вы попадете сразу после создания VPS.

## 🐧 Установка (Ubuntu) ##

---
Данный метод установки не говорит вам, что у вас обязательно должен быть beget VPS. Он может располагаться на любом другом хостинге, главное, чтобы на вашем VPS стоял Ubuntu (чаще всего это именно Ubuntu) 

Ваша задача выгрузить файлы и запустить main.py. Как этого добиться? - достаточно просто.

VPS к этому моменту уже должен закончить свою конфигурацию, поэтому берем наш ранее скачанный репозиторий и закидываем его полностью под root (в папку root) через файловый менеджер VPS. Как только все файлы окажутся на VPS - можно открывать консоль. Самое первое - нужно перейти в директорию с файлами нашего бота, но для начала убедимся, что файлы действительно загружены. Пропишем следующую команду:

```bash
ls
```

Если в списке директорий есть TOASTER - все в порядке, можно переходить в нее:

```bash
cd TOASTER
```

И так, мы в директории бота. Так как в Ubuntu по стандарту поставляется python3 и утилита pip - нам остается лишь установить все нужные библиотеки при помощи команды:

```bash
pip install -r requirements.txt
```

Ждем окончания загрузки и.... готово! Теперь у нас есть все, чтобы запустить бота.
Для этого, оставаясь в той же директории, пропишем команду:

```bash
python3 main.py
```

Готово! Ваш бот работает. Это было не сложно, да? Можете смело переходить к разделу [Быстрый старт](#%EF%B8%8F-быстрый-старт) и продолжать вводить бота в эксплуатацию.

Дополнение: Для того, чтобы остановить бота, достаточно использовать сочетание клавиш `ctrl + c` в консоли, это вызовет keyboard interrupt. Чтобы снова запустить бота - нужно перейти в директорию TOASTER и снова использовать команду, описанную выше.

Для удобства использования Ubuntu можно установить [tmux](https://github.com/tmux/tmux), это позволит создать несколько сессий, что даст возможность запустить несколько ботов.

```bash
apt install tmux
```

В будущем, если вы захотите обновить бота, то просто скачайте базу данных и вставьте ее в скачанный обновленный репозиторий по следующему пути:

```bash
TOASTER-2.0.x/database
```

После чего вы снова можете спокойно по алгоритму, описанному выше, запускать бота.

## 🐋 Установка (Docker) ##

---
Данный метод установки не говорит вам, что у вас обязательно должен быть beget VPS. Он может располагаться на любом другом хостинге, главное, чтобы на вашем VPS c Ubuntu (чаще всего это именно Ubuntu) стоял Docker.

Все таки вам стоить учесть, что, выбрав этот способ, вас может ожидать неприятный [нюанс](#-нюанс).

Ваша задача собрать образ и запустить контейнер. Как этого добиться? - достаточно просто.

VPS к этому моменту уже должен закончить свою конфигурацию, поэтому берем наш ранее скачанный репозиторий и закидываем его полностью под root (в папку root) через файловый менеджер VPS. Как только все файлы окажутся на VPS - можно открывать консоль. Самое первое - нужно перейти в директорию с файлами нашего бота, но для начала убедимся, что файлы действительно загружены. Пропишем следующую команду:

```bash
ls
```

Если в списке директорий есть TOASTER - все в порядке, можно переходить в нее:

```bash
cd TOASTER
```

Как только вы успешно попадете в директорию - нам понадобится создать образ. Для этого пропишем следующую команду:

```bash
docker build . -t toaster 
```
Дожидаемся окончания загрузки и все! Теперь мы имеем опорный образ с именем "toaster", от которого мы сделаем контейнер Docker'a. Убедимся, что мы действительно создали образ:

```bash
docker images
```

Если в списке есть образ с именем "toaster", то вы все еще на верном пути. Далее запустим контейнер:

```bash
docker run --name toaster_cont -d toaster
```
И опять же проверим, все ли прошло гладко:

```bash
docker ps -a
```

Если в списке вы видите контейнер с именем "toaster_cont" - все прошло как надо! Обратите внимание на графу "status", там должно быть написано Up, если Exited - вы создали контейнер, но он у вас выключен. Включить или выключить контейнер можно следующими командами:

   docker stop toaster_cont   # Выключить контейнер toaster_cont
   docker start toaster_cont   # Включить контейнер toaster_cont

**Поздравляю!** Теперь у вас есть работающий бот! Можете смело переходить к разделу [Быстрый старт](#%EF%B8%8F-быстрый-старт) и продолжать вводить бота в эксплуатацию.

### Нюанс ###
В дальнейшем вам может понадобиться пересобрать и перезапускать контейнер по описанному выше алгоритму, если вы вдруг захотите сами обновить или скачать обновления для TOASTER. Это требует более продвинутых знаний и усидчивости, поэтому способ может вам не подойти, так как перед процессом переборки требуется сохранить (выкачать базу данных).  Для этого нужно сделать следующее:

```bash
docker cp toaster_cont:/root/TOASTER/database/database.db copied_base.db 
```

Только что мы скопировали базу данных из контейнера в root директорию VPS. Она называется "copied_base.db". Теперь нам остается скачать эту базу данных через файловый менеджер, а после поместить в скачанный обновленный/измененный репозиторий по следующему пути :

```path
/TOASTER-2.0.x/database
```

Прекрасно. Теперь вы можете удалить контейнер, а затем и образ, а затем снова собрать их по алгоритму, который описан выше:

```bash
docker rm toaster_cont   # Удаляет контейнер
````
```bash
docler rmi toaster        # Удаляет образ
```

## ⬆️ Быстрый старт ##

---
Как только бот оказался на хосте, а в ваших беседах есть корневое сообщество - он готов к использованию.
Так как ранее мы указывали id главного админа в config - вы можете делать все, что пожелаете. (Если, конечно, вы указали свой id)

Далее нужно указать какая беседа будет являться чатом, а какая лога. Первым делом пометим лог-чат при помощи следующей команды:

```text
   /log
```

После чего перейдем в другу беседу и пометим ее как чат, но уже вот этой командой:

```text
   /chat
```

Кстати, если вы вдруг перепутали, какая из бесед будет логом, а какая чатом - вы можете всегда откатить пометку, используя следующую команду:

```text
  /drop
```

Поздравляю! Вы получили первые логи и готовый кластер бесед... В котором, пока что, всего две беседы. К вашему счастью - логов и чатов можно создавать сколько угодно! Логи будут отправлены в каждый лог-чат, а кол-во обычных чатов не ограничено.

Однако... Вы и в правду собираетесь занимать модерированием в одиночку? Позовите своих самых верных друзей и раздайте им права доступа! Делается это при помощи следующей команды:

```text
   /permission 1 @user
```

Или так, но с пересланным сообщением цели:

```text
  /permission 1
```

Теперь вы имеете полностью модерируемый кластер бесед, который вы можете настроить под себя и нужды вашей авантюры, а ручной модерацией будут заниматься ваши друзья, удобно, да? Все действия будут логироваться, а сообщения - тщательно проверяться!

## ⚙ Команды ##

---
 Общие правила работы команд:
   - Команда удаляется в случае выполнение\невыполнения
   - Не всякое сообщение, начинающееся с префикса, считается командой
   - В случае выполнения команда отправляет лог
   - Команда не выполняется в случае отсутствия аргументов в строгом количестве
   - Команды должны вызваться в том же чате, в котором находится целевой пользователь
   - Главный админ может применить любую команду вне зависимости от группы прав
   - Команды, внутри которых отсутствует конструкция @user, подразумевают использование на пересланном сообщении целевого пользователя
--- 
**/log**

- Доступные префиксы: `!` или `/` 
- Псевдонимы команды: ***log***, ***лог*** 
- Доступ для группы прав `2` уровня или выше 

Описание:
Команда помечает беседу, как лог-чат. Теперь в эту беседу будут приходить логи исполненных команд.

---    
**/chat**

- Доступные префиксы: `!` или `/` 
- Псевдонимы команды: ***chat***, ***чат*** 
- Доступ для группы прав `2` уровня или выше 

Описание:
Команда помечает беседу, как чат. Теперь в этой беседе будет проходить модерация фильтрами. 

---    
**/drop**

- Доступные префиксы: `!` или `/` 
- Псевдонимы команды: ***drop***, ***уронить*** 
- Доступ для группы прав `2` уровня или выше 

Описание:
Команда сбрасывает метку чата или лог-чата с беседы.

---
**/permission** `<lvl>` `<@user|optional>`

- Доступные префиксы: `!` или `/` 
- Псевдонимы команды: ***permission***, ***perm***,  ***права***
- Доступ для группы прав `2` уровня или выше 
- Может быть вызвана в лог-чате

Описание:
Команда устанавливает для пользователя группу прав, равную введенному аргументу. 

Доступные аргументы:
- `<lvl>`: `0` - user, `1` - moderator, `2` - administrator

---   
**/reference**

- Доступные префиксы: `!` или `/`
- Псевдонимы команды: ***reference*** , ***ref***, ***справка***
- Доступ для группы прав `1` уровня или выше
- Может быть вызвана в лог-чате

Описание:
Выводит в чат ссылку на этот README.md, если кто вдруг забыл какую-то команду.

---
**/delete**

- Доступные префиксы: `!` или `/`
- Псевдонимы команды: ***delete*** , ***del***, ***удалить***
- Обработка только совместно с пересланным сообщением
- Доступ для группы прав `1` уровня или выше
- Может быть вызвана в лог-чате

Описание:
Удаляет пересланное или группу пересланных сообщений.

---
**/copy**

- Доступные префиксы: `!` или `/`
- Псевдонимы команды: ***copy***, ***копировать***
- Обработка только совместно с пересланным сообщением
- Доступ для группы прав `1` уровня или выше
- Может быть вызвана в лог-чате

Описание:
Копирует текст пересланного сообщения и отправляет в беседу от лица бота.

---
**/terminate** `<@user|optional>`

- Доступные префиксы: `!` или `/`
- Псевдонимы команды: ***terminate***, ***term***, ***терминировать***
- Доступ для группы прав `2` уровня или выше
- Может быть вызвана только в чате

Описание:
Исключает пользователя из всех бесед навсегда. 

---
**/kick** `<@user|optional>`

- Доступные префиксы: `!` или `/`
- Псевдонимы команды: ***kick***, ***кик***
- Доступ для группы прав `1` уровня или выше
- Может быть вызвана только в чате

Описание:
Исключает пользователя из беседы навсегда. 

---
**/ban** `<time>` `<coefficent>` `<@user|optional>`

- Доступные префиксы: `!` или `/`
- Псевдонимы команды: ***ban***, ***бан***
- Доступ для группы прав `1` уровня или выше
- Может быть вызвана только в чате

Описание:
Блокирует пользователя на некоторый промежуток времени и удаляет из чата. Уведомление об окончании блокировки поступит в лог-чаты. 

Доступные аргументы:
- `<time>`: натуральное число
- `<coefficent>`: `h` - hour, `d` - day, `m` - month

## ⚠️ Настройки ##

---
Стандартные настройки для беседы:
1. `Allow_Picture` - Изображения
2. `Allow_Video` - Видео
3. `Allow_Music` - Музыка
4. `Allow_Links` - Ссылающее вложение
5. `Allow_Voice` - Голосовое сообщение
6. `Allow_Post` - Репост/запись в чат
7. `Allow_Votes` - Голосования/опросы
8. `Allow_Files` - Файлы
9. `Allow_Miniapp` - Мини-приложения
10. `Allow_Graffit` - Файлы
11. `Allow_Sticker` - Стикеры
12. `Allow_Reply` - Пересланные сообщения
13. `Filter_Curse` - Нежелательные слова
14. `Slow_Mode` - Медленный режим
15. `Account_Age` - Возраст аккаунта
16. `Hard_Mode` - Режим фильтрации ссылок

Каждая из этих настроек отвечает за свой тип контента, который разрешен в данной беседе. По стандарту все значения настроек установлены на ___False___. Это можно изменить в разделе SETTINGS внутри config.py. Разберем по порядку каждую настройку:

- `1-11`: Данная группа настроек отвечает за вложенный контент в сообщениях. <br> 
**True** - фильтр не проверяет сообщения на наличие вложенного контента данного типа.<br> 
**False** - фильтр проверяет сообщение на наличие вложенного контента данного типа.<br> 
В случае обнаружения - пользователю выдается предупреждение.
Для понимания какого рода контент будет пресекаться - см. описание настроек выше.

-  `12`: Настройка, отвечающая за проверку на пересланные сообщения. <br> 
**True** - отлов пересланных сообщений включен. <br> 
**False** - отлов пересланных сообщений выключен. <br> 
Вы можете запретить людям пересылать чужие сообщения, а так же отвечать на сообщения. Если фильтр зафиксирует нарушение - пользователь получит предупреждение.
- `13`: Настройка, отвечающая за проверку наличия в сообщениях нежелательных слов. <br> 
**True** - фильтр нежелательных слов включен. <br> 
**False** - фильтр нежелательных слов выключен. <br> 
Если фильтр обнаружит такие слова в сообщении - сообщение будет удалено. Пользователь не получит никакого наказания, но поймет, что такие слова не приветствуются.

- `14`: Настройка, включающая или выключающая медленный режим в беседе. <br> 
**True** - медленный режим включен. <br> 
**False** - медленный режим выключен. <br> 
Если пользователь напишет сообщение при включенном медленном режиме, то его поместит в очередь. Из этой очереди он будет убран только по истечении определенного времени, которое можно установить параметром **QUEUE_TIME** в **config.py**. Время указывается в секундах.
Если пользователь напишет сообщение, но он уже был в очереди медленного режима, то пользователю будет выдано предупреждение.

- `15`: Настройка, включающая или выключающая проверку возраста аккаунта. <br> 
**True** -  проверка возраста аккаунта включена. <br> 
**False** - проверка возраста аккаунта выключена. <br> 
Если пользователь напишет сообщение в чат - его аккаунт будет проверен на соответствие к возрастной планке. Проверяется дата создания аккаунта и, если эта дата, была менее чем две недели назад, то пользователь будет исключен из беседы. Данный фильтр создан для защиты от спама роем ботов.

- `16`: Настройка, включающая или выключающая усиленный режим проверки ссылок. <br> 
**True** - усиленная проверка включена. <br> 
**False** - усиленная проверка выключена. <br> 
По стандарту бот всегда проверяет наличие ссылок в сообщении. Если настройка выключена, то бот выдаст заглушение только тогда, когда ссылка или домен этой ссылки входит в категорию **CRITICAL** внутри **config.py**. Если настройка включена, то помимо заглушения за ссылки категории **CRITICAL**, бот выдаст предупреждение за любую ссылку или домен этой ссылки, который не находится в категории **ALLOWED** внутри **config.py**.
