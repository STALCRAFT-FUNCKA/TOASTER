## 📄 Введение ##

**TOASTER** - кластерный чат-бот для социальной сети VK.
Он был создан и спроектирован для работы штата модерации в кластере бесед неформального фанатского сообщества [FUNCKA](https://vk.com/funcka) по игре [STALCRAFT](https://vk.com/stalcraft_official). Проект является полностью opensource, что значит - вы сможете использовать бота на своих кластерах бесед.

## ⚙️ Настройка VPS ##

**ВНИМАНИЕ:** Для работы бота требуется новое материнское сообщество. Просто создайте сообщество по шаблону "интересы". Затем включите сообщения для этого сообщества и функцию добавления сообщества в другие чаты. Добавьте кнопку добавления сообщества в левое меню сообщества.

Все эти действия можно сделать, перейдя:

> Сообщество -> Управление -> Сообщения\Настройки бота

В данном пункте будет находиться краткая, но емкая информация касательно установки TOASTER на хостинг.

Для начала нам нужно арендовать VPS с Ubuntu 22.04, на котором установлен docker.Воспользоваться можно любым хостингом, но для чаников рекомендуется [Beget](https://beget.com/ru). Там можно выбрать готовый пресет VPS с Docker. А теперь о минимальных системных требованиях для VPS:

* 1 CPU Core
* 2 GB RAM
* 10 GB MEM

Напомню, что требования минимальны. Старайтесь рассчитать их так, как того требует ваша ситуация. Учтите то, сколько у вас бесед, суммарное кол-во пользователей, а так же кол-во сообщений в секунду. Для средних и уверенных настроек рекомендую что-то вроде:

* 2 CPU Core
* 4 GB RAM
* 20-25 GB MEM

Теперь перейдем к самому интересному. Нам необходимо:

* Развернуть контейнер MySQL
* Развернуть контейнер TOASTER

Начнем с первого пункта. Открываем консоль VPS и прописываем следующую команду:

```shell
docker run --name mysql_cont \
-p 3306:3306 \
-e MYSQL_ROOT_PASSWORD=your-password \
-d mysql
```

Данная команда скачает с Dockerhub последнюю версию MySQL Server и завернет его в контейнер. Не забудьте указать свой пароль для БД, а так же запишите его куда-нибудь, он нам еще понадобится.

Для уверенности, можно прописать:

```shell
docker ps -a
```

Если в списке есть mysql_cont со статусом Up, то вы все сделали верно. Если же у вас не так и вы накосячили - удалите docker container и повторите предыдущий шаг с большей внимательностью.

И так, база данных развернута, теперь переходим к развертке TOASTER:

```shell
git clone https://github.com/STALCRAFT-FUNCKA/TOASTER
```

Тут нам нужно собрать образ TOASTER. Ниже будет представлена команда, которая как раз таки и исполнит данное действие, однако! Перед тем, как ее запустить, давайте создадим переменные окружения. В них мы поместим все необходимые для запуска данные.

Шаблон, который находится ниже, нужно заполнить данными. Исключительно в тех местах,
где есть пометка example. Что ж, открываем .bashrc:

```shell
sudo nano .bashrc
```

В открывшийся фал, прямо в его конец, вставляем следующие строки:

```shell
export TOASTER_TOKEN="your_token"   # example
export TOASTER_GROUPID=2041294120   # example
export TOASTER_STAFFADMID=4124124   # example

export SQL_HOST="255.255.255.255"   # example
export SQL_PORT=3306                
export SQL_USER="root"              
export SQL_PASSWORD="your-db-pwd"   # example
```

Повторное предупреждение: Изменяемыми тут будут только те данные, которое помечены example

**Где же брать нужные нам данные?**

 * **TOASTER_TOKEN** <br>
  Это токен доступа бота к вашей группе. Его нужно создать в настройках материнского сообщества в разделе "Работа с API". Создаем токен (он же ключ доступа), выдавая ему все права (указывая все галочки). Далее переходим в раздел "LongpollAPI", включаем его. Выставляем все галочки в разделе "сообщения", выбираем версию v5.154. Токен, который уже приобрел все настройки, вставляем в нужно место внутри .bashrc
 
 * **TOASTER_GROUPID** <br>
  Это буквально id материской группы. Узнать его можно в настройках сообщества, графа "Адрес"
 
 * **TOASTER_STAFFADMID** <br>
  Это буквально id пользователя, которого вы хотите сделать главным админом. Данный человек может использовать любые команды, вне зависимости от уровня прав доступа.

 * **SQL_HOST** <br>
  Это локальный ip mysql, чтобы найти его, пропишите команду, как указано ниже, скопируйте ip и вставьте в .bashrc:

  ```shell
  docker inspect mysql_cont | grep "IPAddress"
  ```

 * **SQL_PASSWORD** <br>
  Это пароль к базе данных, который был указан ранее, при поднятии контейнера mysql.

Как только файл .bashrc оказывается отредактирован должным образом - сохраняем его и забываем. Далее перезапускаем консоль.

Теперь можно наконец таки собрать образ TOASTER и поднять контейнер.

```shell
cd TOASTER/
```

Переходим в директорию репозитория. Прописываем команду, которая запустит построение образа:

```shell
docker build . -t toaster_img \
--build-arg TOKEN=$TOASTER_TOKEN \
--build-arg ID=$TOASTER_GROUPID \
--build-arg ADM=$TOASTER_STAFFADMID \
--build-arg HOST=$SQL_HOST \
--build-arg PORT=$SQL_PORT \
--build-arg USER=$SQL_USER \
--build-arg PASSWORD=$SQL_PASSWORD
```

Дальше можно запускать контейнер:

```shell
docker run -d --name toaster_cont toaster_img
```

Готово! Ваш бот успешно встал на рельсы. Следом нужно добавить его в беседы.

Вы можете создать беседы от лица сообщества, а можете пригласить бота в другие беседы. Главное условие: 
минимальное кол-во бесед - 2 штуки.
  
## ⬆️ Быстрый старт ##

Как только бот оказался на хосте, и в ваших беседах - он готов к использованию. Так как ранее мы указывали id главного админа - он и должен выполнить следующие шаги.

**ВАЖНО**: Не рекомендуется использовать бота, имея VKadmin в беседе. Вместо этого выдайте доступ
до всех сообщений беседы боту, а так же VKadmin.

Далее нужно указать какая беседа будет являться чатом, а какая лога. Первым делом пометим лог-чат при помощи следующей команды:

```text
/mark log
```

После чего перейдем в другую беседу и пометим ее как чат, но уже вот этой командой:

```text
/mark chat
```

Кстати, если вы вдруг перепутали, какая из бесед будет логом, а какая чатом - вы можете всегда откатить пометку, используя следующую команду:

```text
/mark drop
```

Поздравляю! Вы получили первые логи и готовый кластер бесед... В котором, пока что, всего две беседы. К вашему счастью - логов и чатов можно создавать сколько угодно! Логи будут отправлены в каждый лог-чат, а кол-во обычных чатов не ограничено.

Однако... Вы и в правду собираетесь занимать модерированием в одиночку? Позовите своих самых верных друзей и раздайте им права доступа! Делается это при помощи следующей команды:

```text
/permission 1 @user
```

Или так, но с пересланным сообщением цели:

```text
/permission 1
```

Теперь вы имеете полностью модерируемый кластер бесед, который вы можете настроить под себя и нужды вашей авантюры, а ручной модерацией будут заниматься ваши друзья, удобно, да? Все действия будут логироваться, а сообщения - тщательно проверяться!

Обратите внимание: Команды, внутри которых отсутствует конструкция `@user`, подразумевают использование на пересланном сообщении другого пользователя.

## ⚙ Команды ##

 Общие правила работы команд:
   - Команда удаляется в случае выполнение\невыполнения
   - Не всякое сообщение, начинающееся с префикса, считается командой
   - В случае выполнения команда отправляет лог
   - Команда не выполняется в случае отсутствия аргументов в строгом количестве
   - Команды должны вызваться в том же чате, в котором находится целевой пользователь
   - Главный админ может применить любую команду вне зависимости от группы прав
   - Команды, внутри которых отсутствует конструкция @user, подразумевают использование на пересланном сообщении целевого пользователя
  
--- 

**/mark** `<arg>`

- Доступные префиксы: `!` или `/` 
- Псевдонимы команды: ***mark***, ***метка*** 
- Доступ для группы прав `2` уровня или выше 

Описание:
Команда помечает беседу лог-чатом или чатом. Так же метку чата можно сбросить

Доступные аргументы:
* `<arg>`: log, chat, drop

---
**/permission** `<lvl>` `<@user|optional>`

- Доступные префиксы: `!` или `/` 
- Псевдонимы команды: ***permission***, ***perm***,  ***права***
- Доступ для группы прав `2` уровня или выше

Описание:
Команда устанавливает для пользователя группу прав, равную введенному аргументу. 

Доступные аргументы:
- `<lvl>`: `0` - user, `1` - moderator, `2` - administrator

---   
**/reference** `<command|optional>`

- Доступные префиксы: `!` или `/`
- Псевдонимы команды: ***reference*** , ***ref***, ***справка***
- Доступ для группы прав `1` уровня или выше
- Может быть вызвана только в лог-чате

Описание: 
Выводит исчерпывающую справку по какой-либо команде.

Доступные аргументы:
- `<command>`: `all` (всеобщая справка) или любая `команда`, включая ее псевдонимы.

---   
**/info** `<info_type>`

Описание:

Доступные аргументы:
- `<info_type>`: mark, setting, permission, kick, ban, mute, warn

Описание:
Удаляет пересланное или группу пересланных сообщений.

## ⚠️ Настройки ##

Стандартные настройки для беседы:
1. `Allow_Picture` - Изображения
2. `Allow_Video` - Видео
3. `Allow_Music` - Музыка
4. `Allow_Links` - Ссылающее вложение
5. `Allow_Voice` - Голосовое сообщение
6. `Allow_Post` - Репост/запись в чат
7. `Allow_Votes` - Голосования/опросы
8. `Allow_Files` - Файлы
9. `Allow_Miniapp` - Мини-приложения
10. `Allow_Graffit` - Файлы
11. `Allow_Sticker` - Стикеры
12. `Allow_Reply` - Пересланные сообщения
13. `Filter_Curse` - Нежелательные слова
14. `Slow_Mode` - Медленный режим
15. `Account_Age` - Возраст аккаунта
16. `Hard_Mode` - Режим фильтрации ссылок
17. `Need_PM` - Обязует иметь открытыое ЛС

Каждая из этих настроек отвечает за свой тип контента, который разрешен в данной беседе. По стандарту все значения настроек установлены на ___False___. Это можно изменить в разделе SETTINGS внутри usr_config.py. Разберем по порядку каждую настройку:

- `1-11`: Данная группа настроек отвечает за вложенный контент в сообщениях. <br> 
**True** - фильтр не проверяет сообщения на наличие вложенного контента данного типа.<br> 
**False** - фильтр проверяет сообщение на наличие вложенного контента данного типа.<br> 
В случае обнаружения - пользователю выдается предупреждение.
Для понимания какого рода контент будет пресекаться - см. описание настроек выше.

-  `12`: Настройка, отвечающая за проверку на пересланные сообщения. <br> 
**True** - отлов пересланных сообщений включен. <br> 
**False** - отлов пересланных сообщений выключен. <br> 
Вы можете запретить людям пересылать чужие сообщения, а так же отвечать на сообщения. Если фильтр зафиксирует нарушение - пользователь получит предупреждение.
- `13`: Настройка, отвечающая за проверку наличия в сообщениях нежелательных слов. <br> 
**True** - фильтр нежелательных слов включен. <br> 
**False** - фильтр нежелательных слов выключен. <br> 
Если фильтр обнаружит такие слова в сообщении - сообщение будет удалено. Пользователь не получит никакого наказания, но поймет, что такие слова не приветствуются.

- `14`: Настройка, включающая или выключающая медленный режим в беседе. <br> 
**True** - медленный режим включен. <br> 
**False** - медленный режим выключен. <br> 
Если пользователь напишет сообщение при включенном медленном режиме, то его поместит в очередь. Из этой очереди он будет убран только по истечении определенного времени, которое можно установить параметром **QUEUE_TIME** в **usr_config.py**. Время указывается в секундах.
Если пользователь напишет сообщение, но он уже был в очереди медленного режима, то пользователю будет выдано предупреждение.

- `15`: Настройка, включающая или выключающая проверку возраста аккаунта. <br> 
**True** -  проверка возраста аккаунта включена. <br> 
**False** - проверка возраста аккаунта выключена. <br> 
Если пользователь напишет сообщение в чат - его аккаунт будет проверен на соответствие к возрастной планке. Проверяется дата создания аккаунта и, если эта дата, была менее чем две недели назад, то пользователь будет исключен из беседы. Данный фильтр создан для защиты от спама роем ботов.

- `16`: Настройка, включающая или выключающая усиленный режим проверки ссылок. <br> 
**True** - усиленная проверка включена. <br> 
**False** - усиленная проверка выключена. <br> 
По стандарту бот всегда проверяет наличие ссылок в сообщении. Если настройка выключена, то бот выдаст заглушение только тогда, когда ссылка или домен этой ссылки входит в категорию **CRITICAL** внутри **usr_config.py**. Если настройка включена, то помимо заглушения за ссылки категории **CRITICAL**, бот выдаст предупреждение за любую ссылку или домен этой ссылки, который не находится в категории **ALLOWED** внутри **usr_config.py**.

- `17`: Настройка, включающая проверку на открытые личные сообщения у пользователя. <br>
**True** - проверка включена. <br>
**False** - проверка включена. <br>
Если пользователь напишет сообщение в чат, а его личные сообщения будут закрыты, то пользователь получит предупреждение, а его сообщение будет удалено.
